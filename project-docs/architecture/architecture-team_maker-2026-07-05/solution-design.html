team_maker — v1 Architecture

The conversational multi-agent team builder. Companion to ARCHITECTURE-SPINE.md (the build contract) and PRD Rev 2. Layered + ports-and-adapters around the existing factory. Updated 2026-07-06.

1. What we're building

A user describes a team of AI "thinkers" in a short back-and-forth; an LLM writes the valid configuration; the system builds the team and runs it (or the user chats with it) while the agents collaborate across multiple model providers. It ships as one open-source repo, installable via Docker, pip, or a desktop bundle. v1 is aimed at semi-technical users who bring their own keys; heavy capabilities (memory, always-on teams, local-model fleets) are deliberately v2.

3. The core pipeline — who owns team design

Composer (LLM) authors the Team Spec  →  Factory deterministically builds the Team Package  →  Runtime (CrewAI) executes only.

The runtime never decides team membership or roles — no CrewAI auto-planning. Composition stays LLM-driven and multi-provider, which is the product's differentiator (AD-5). Note: today's factory is template-driven; the LLM composition is the new Composer.

5. The multi-provider risk (verified) & how we contain it

Verified: CrewAI 1.12.x had a multi-provider routing bug (GH #5139) — agents on different providers fell back to global env credentials, so two providers couldn't be served at once. Closed "not planned". Latest is 1.14.6.

Containment: (1) explicit per-agent LLM instances with full credentials — never global env; (2) a required multi-provider conformance test that gates any CrewAI upgrade; (3) runtime behind a port so a swap is contained; (4) OpenRouter promoted to the default path — one key + one endpoint sidesteps the whole bug class. Version pin of 1.14.6 is held open until the conformance test confirms per-agent routing.

6. The v1 → v2 seam

In v1

  • Conversational Composer + validate-and-repair
  • Multi-provider factory + OpenRouter
  • CrewAI runtime; agents collaborate; batch results
  • Team Workspace: chat + transient docs + save
  • Recent-teams list; starter education + research/content teams
  • Local SQLite/files; keys in a file

In v2 (attach as adapters)

  • Team memory / learning → Memory port
  • Always-on teams → Lifecycle port
  • Managed local-model fleets
  • Streaming results → AD-13 interface
  • Team versioning; hosted/no-keys tier
  • ~20 domain team templates; desktop wrapper
