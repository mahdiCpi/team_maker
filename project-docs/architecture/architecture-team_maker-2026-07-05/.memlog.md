Distribute via multiple channels: docker image, pip (Python core + CLI), desktop bundle, web UI served by the backend. Prevents: multi-repo/microservice sprawl that users can't easily download/install. (user: one repo, easy install via docker/npm/etc.)
- 2026-07-05 | constraint | Monorepo holds two languages: Python backend/core (existing factory + Composer + Runtime + API) and TS/React frontend (Next.js + shadcn per UX). Single repo, workspace layout.
- 2026-07-05 | decision | Runtime engine = CrewAI, behind a Runtime PORT (option A). Prevents: single-provider lock-in (core wedge) and hard coupling to one engine (swap = adapter change). (user chose A)
- 2026-07-05 | decision | Team-composition OWNERSHIP: the Composer (LLM, L2) authors the Team Spec → Factory deterministically builds the Team Package → Runtime (CrewAI) EXECUTES ONLY. The runtime never decides team membership/roles; CrewAI auto-planning/hierarchical auto-assignment is NOT used. Prevents: CrewAI silently owning composition and eroding the LLM-composed, multi-provider differentiator. (addresses user concern)
- 2026-07-05 | constraint | Clarification: today's factory is TEMPLATE/CONFIG-driven (deterministic _ROLE_DEFAULTS + user config), NOT LLM-driven. The LLM-driven composition is the NEW Composer being added in v1. (corrects "team_maker uses an LLM to create teams")
- 2026-07-05 | version | CrewAI latest = 1.14.6 (PyPI). Uses native SDKs for OpenAI/Anthropic/Gemini/Azure/Bedrock + LiteLLM fallback for others; supports Ollama (local). Model strings like anthropic/claude-sonnet-4-*, etc.
- 2026-07-05 | question | VERIFIED RISK (user's memory confirmed): CrewAI 1.12.x had a multi-provider routing bug (GH #5139) — LLM stored base_url but litellm expects api_base, and agent-internal calls bypass per-LLM creds and fall back to GLOBAL env vars, so two agents on different providers can't be served simultaneously. Closed "not planned". Also #2220 litellm BadRequestError. Must confirm whether pinned version (1.14.6) fixes per-agent multi-provider routing BEFORE binding.
- 2026-07-05 | decision | Multi-provider routing mitigations (become ADs): (a) pin a verified CrewAI version; (b) construct explicit per-agent LLM instances with full per-agent credentials — never rely on global env for routing; (c) a required multi-provider CONFORMANCE TEST: a team spanning >=2 providers asserts each agent hit its intended provider; gate CrewAI upgrades behind it; (d) keep runtime behind the port so a swap is contained.
- 2026-07-05 | decision | OpenRouter is elevated from convenience to RISK-MITIGATION: routing all agents through one OpenRouter key collapses multi-provider into a single-credential/single-endpoint problem, sidestepping the env-var-fallback bug class. Recommend OpenRouter as the default path for users who don't want to manage many keys.
- 2026-07-06 | decision | Composer sits behind the SAME LLM/Provider port (swappable, OpenRouter-capable) with a configurable default model; emits the Team Spec via structured output VALIDATED against the factory's Pydantic schema, with a bounded validate-and-repair loop (FR-2). Prevents: the Composer becoming a hard-coded/second LLM integration path or emitting invalid specs. (user approved)
- 2026-07-06 | decision | State & storage = LOCAL, embedded, ZERO external services in v1: SQLite + files for recent-teams list and saved results; Team Packages written to disk by the factory (self-contained); Key Config is a user-managed file the backend only READS; documents are TRANSIENT per-run (loaded into run context, not persisted). Memory and always-on ports exist but are unimplemented in v1. Prevents: infra dependence that breaks the "download and run" promise; premature DB/cloud coupling. (user approved)
